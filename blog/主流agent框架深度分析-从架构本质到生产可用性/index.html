主流 Agent 框架深度分析：从架构本质到生产可用性

本文综合了对 Hermes Agent 开源代码库的架构分析，以及在企业级 AI 安全平台（AEGIS）设计过程中对 9 个主流框架的系统性选型评估。两个视角相互补充：前者是”框架作者视角”，后者是”框架消费者视角”。

一、为什么框架选型如此困难

AI Agent 框架的选型是一道比它看起来更难的题。

大多数框架的 README 都能让你在 30 分钟内跑起一个 Demo，但真正的问题要 3 个月后才会暴露：上下文管理撑不住生产规模、多用户并发没有会话隔离、权限控制是事后打补丁而非架构原生、工具数量膨胀后模型选择准确率开始下降……

这些问题有一个共同根因：多数 Agent 框架是为原型验证设计的，不是为生产系统设计的。

更准确地说，框架作者在设计时面临一个两难困境：通用性和生产可用性往往是反向的。越通用的框架越难做到开箱即用的生产就绪；越垂直的框架越适合特定场景，但走出这个场景就会遇到天花板。

因此本文不做”功能对比矩阵”，而是从架构本质出发，回答三个问题：

  1. 每个框架的核心设计抽象是什么？
  2. 这个抽象在哪些场景下成立，在哪些场景下会崩溃？
  3. 如果要基于它构建生产系统，你会踩进哪些坑？

二、框架全景：三种产品形态

按运行形态，当前主流框架分为三类，这个分类比”功能有没有”更根本——形态决定了适用场景的上限。

本地工具型代表有：Claude Code、Cursor、OpenAI Codex CLI。共同特征是单用户、本地 CLI，面向开发者。设计理念成熟，工程细节经过打磨，但无法直接用于服务端多用户场景。

通用编排库代表有：LangGraph、CrewAI、AutoGen。共同特征是 Python 库，可包装为微服务，API 化程度高。灵活性最强，但生产就绪程度参差不齐，通常需要大量定制才能进入生产。

平台型代表有：Dify、Coze、Hermes。共同特征是完整系统，含前端和管理界面，功能生态最完整。但难以嵌入现有系统，适合作为独立产品使用。

这个分类的关键洞察：产品形态不匹配不等于没有参考价值。 Claude Code 是架构设计最成熟的 Agent 系统之一，但它是本地 CLI，无法直接用于多用户服务端。Hermes 是功能最完整的开源平台，但它是完整产品，不是可嵌入的引擎框架。两者都有极高的学习价值，只是”不能直接用”。

三、本地工具型：架构设计的天花板

Claude Code（Anthropic）

定位：面向开发者的 AI 编程 Agent，本地 CLI，TypeScript/Bun 实现。

核心架构模型：Agent 循环（核心）= LLM Call → Tool Selection → 权限评估 → Tool Execute → 结果回传，循环迭代。

这个核心循环之上，内建了若干关键机制：

  • CLAUDE.md：项目级上下文文件，Agent 启动时加载，实现对当前代码库的”感知”
  • Hook 体系：15+ 生命周期事件，通过 AOP 模式拦截，外部逻辑不侵入内核
  • Compaction（含断路器）：上下文超限时自动压缩，压缩后仍超限则终止而非无限重试
  • ToolSearch（三档加载）：高频工具常驻上下文、中频工具按需加载、低频工具休眠
  • Prompt Cache：会话内 System Prompt 不重建，保证缓存有效性
  • SubAgent / Teammate / Task：三级委托模型，支持任务拆分与并行

为什么说它架构最成熟

Claude Code 经历了日活百万级用户的生产验证，它的每一个设计决策背后都有真实的失败案例支撑：

Compaction 带断路器，是因为曾经出现过连续压缩仍超限导致无限重试的线上事故；ToolSearch 三档加载，是因为工具膨胀到一定数量后模型的工具选择准确率会显著下降；Hook 体系而非侵入式修改，是因为审计、权限、监控如果耦合进 Agent 循环内核，维护成本会指数级上升；Prompt Cache 作为硬约束，是因为 LLM API 调用成本在大规模使用下不可忽视。

每一个设计决策都有”为什么不是更简单的方案”的答案。这是从生产失败中提炼的工程智慧，是 Demo 阶段永远无法获得的。

适用场景：个人或团队开发辅助，IDE 集成，代码自动化

不适用场景：多用户服务端（单用户设计），私有化部署（本地 CLI 形态），安全敏感系统（无多租户隔离）

OpenAI Codex CLI

核心模式：统一 App Server + CodeAct（代码即行动）

CodeAct 的设计哲学值得单独讨论：与其为 Agent 设计一套工具 API，不如让 Agent 直接写代码来执行。这个哲学消除了”工具设计”这个需要持续维护的层，理论上 Agent 可以做任何可以用代码表达的事情。

但这个优雅性的代价是：沙箱隔离和安全边界变得更难保障。当工具调用是结构化的 JSON 时，你可以精确控制每个参数；当”工具调用”变成任意代码时，你需要更复杂的沙箱机制来防止越界行为。

CodeAct 是一个有趣的架构取舍——用工具设计的复杂性换取沙箱安全的复杂性。哪个更难管理，取决于你的团队和场景。

适用场景：代码生成与执行，轻量本地自动化

不适用场景：需要结构化工具调用、审批流程或多用户的场景

Cursor

核心模式：VS Code Fork + Composer（多文件编辑协调）

Cursor 的特殊之处在于它把 Agent 深度嵌入了编辑器状态——Git 历史、打开的文件、光标位置、选中代码块都是上下文的一部分。这是典型的”深度垂直整合”路线，在代码编辑场景内效果极佳。

这种上下文模型的价值在于：它不需要 Agent 主动”读取”代码文件，因为编辑器本身就是上下文的容器。Agent 看到的世界和开发者看到的世界在同一个界面内同步。

这个设计的局限也很清楚：这种上下文模型只对编辑器有意义，一旦离开编辑器场景，整个上下文优势就消失了。这是垂直整合的经典取舍——在场景内极强，在场景外无用。

适用场景：纯代码编辑辅助，IDE 内多文件重构

不适用场景：编辑器之外的任何场景

四、通用编排库：生产可用性的真实差距

LangGraph（LangChain 旗下）

核心抽象：状态图 DAG（有向无环图）+ 检查点恢复

LangGraph 的架构可以用一个简单的模型描述：工作流是一张图，图中每个节点是一个处理步骤，边是条件分支。每个节点执行后自动创建检查点，任务失败可以从任意检查点恢复。Human-in-the-loop 是图中的一个特殊节点，有完整的暂停和恢复语义。

真正的亮点：Human-in-the-loop 是一等公民。

LangGraph 是少数几个把”人工审批”设计到框架骨架里的框架——不是作为回调，而是作为图中一个正式节点。这意味着”等待人工审批”是一种正常的流程状态，不是异常路径。在需要审批流的业务场景（如财务审批、安全变更、合规操作）中，这个设计非常有价值。

被高估的地方：状态图 DAG 适合固定工作流，不适合开放式推理。

当 Agent 的每轮工具选择不可预测时，用 DAG 建模会变成一件荒诞的事——你在试图把不确定性装进确定性的盒子里。如果你的 Agent 需要根据当前信息动态决定下一步做什么，那么预定义的图结构只会成为约束，而不是帮助。

真实项目中经常出现的困境是：业务 PM 要求在工作流图上加一个分支，但这个分支的触发条件依赖于 LLM 的输出，而 LLM 的输出是不确定性的。这时你面临一个选择：要么把不确定性硬编码为确定性条件（导致行为过度刚性），要么引入一个专门的”决策节点”（让图变得越来越复杂）。

生产使用的真实挑战

LangGraph 缺少生产系统的几个关键能力：多用户会话隔离（面向单用户或小团队设计）、原生权限控制体系（“谁能调用什么工具”不是框架数据结构）、合规级审计日志（可选配置而非架构约束），以及大型上下文的精细化管理。

适合：有明确流程图的工作流自动化，需要审批节点的 RPA 类场景，状态机管理复杂的 Agent 任务

不适合：开放式推理型 Agent，多用户 SaaS，安全敏感系统

CrewAI

核心抽象：角色化 Agent 团队 + Manager 分派

CrewAI 的模型非常直观：你有一个 Manager Agent（协调者）和若干 Worker Agent（每个有独立角色）。Manager 把任务分派给最合适的 Worker，Worker 在自己的专业领域内执行，然后把结果汇报给 Manager。

真正的亮点：角色化分工的直觉极好。

把复杂任务拆分给专业化的子 Agent，让每个 Agent 的上下文更聚焦、工具集更精简，是应对”工具膨胀”问题的一个优雅思路。一个专注于”信息检索”的 Agent，其上下文里只有检索相关的工具，比一个上下文里塞了 40 个工具的全能 Agent 更不容易出错。

被高估的地方：CrewAI 本质上是一个轻量原型框架，被过度宣传了。

它缺少上下文管理、权限控制、审计、生命周期钩子等生产就绪能力。这不是技术实现问题，而是设计目标问题——CrewAI 的设计目标是”让多 Agent 协作变得简单”，不是”让多 Agent 协作变得可靠”。

这两个目标在原型阶段是兼容的，在生产阶段会开始分叉。

生产使用的真实挑战

上下文管理是最小化的，撑不住真实业务规模的工具清单；没有权限控制、审计、生命周期钩子，这些在生产系统中是刚需；长期演进风险较高，因为框架本身在这些方向上的演进路径不清晰。

适合：快速验证多 Agent 协作思路，教学演示，原型验证

不适合：生产级多用户系统，安全要求高的场景，需要长期维护的系统

AutoGen（微软）

核心抽象：事件驱动消息传递 + 异步拓扑

AutoGen 的设计思路与 CrewAI 有本质差异：Agent 之间不是函数调用，而是消息传递。消息有明确的类型和路由，Agent 通过订阅特定类型的消息来接收任务，通过发布消息来输出结果。

真正的亮点：消息传递架构的扩展性最好。

Agent 之间不是函数调用意味着：Agent 可以跨进程、跨机器、异步协作，拓扑完全可重组。你可以轻松实现”把某个 Agent 部署在算力更强的机器上”或”把某个 Agent 替换为更新版本”而不影响其他 Agent，因为它们之间只通过消息耦合。

被高估的地方：这种架构复杂度对大多数场景是过度设计。

如果你的 Agent 都运行在同一个进程里，引入消息总线反而增加了心智负担——原来一行函数调用解决的事情，现在需要定义消息类型、注册路由、处理异步回调。复杂性的代价要在真正需要分布式协作时才能收回成本。

最大的现实问题：AutoGen 正在向 AG2（AutoGen 0.4）迁移，核心 API 重写中。

在一个核心框架重构期间把它选为生产系统的基础，等于主动引入了”跟踪上游重写”的维护负担。这不是技术问题，是项目风险管理问题。

适合：分布式多 Agent 场景，研究型 Agent 系统，需要高度可重组拓扑的场景

不适合：生产时间线紧张的系统，单进程内的多 Agent 协作，团队没有异步架构经验的场景

通用编排库的共同盲点

三个框架共同缺失的能力，是它们作为通用框架在设计上无法解决的，因为这些需求超出了它们的设计目标：

多用户会话隔离：三个框架都是面向单用户或小团队设计的，没有原生的多租户会话隔离。你可以通过在外层包装隔离逻辑来绕过，但这不是框架原生支持的，意味着你在承担这部分的设计和维护责任。

信任层级原生化：权限是配置，不是引擎数据结构。“谁能调用哪个工具、在什么条件下、经过什么审批”这类问题，这三个框架都没有原生的答案，需要你自己构建。

合规级审计：审计日志可选，不是架构约束。在 SOC2、ISO27001 等合规场景下，审计是必须的而非可选的。把可选变成必须，需要框架层面的支撑，而不只是加一个 logging 库。

大规模工具管理：当工具数量超过 100 个时，如何保证模型的工具召回准确率？三个框架都没有系统性的答案，而 Claude Code 的 ToolSearch 三档加载是目前最成熟的解法。

领域感知上下文：这些框架不理解特定业务系统的领域概念，需要你自己构建”Agent 如何理解当前业务状态”的上下文管理层。

这不是框架的”缺陷”，而是设计目标不同。通用框架做的是”可能性空间”，领域系统做的是”约束空间”。两者之间的 gap，用定制代码填不满——填到一定程度，维护成本不低于自研。

五、平台型：完整产品的两面

Dify

核心模式：可视化工作流 DAG + 多模型 + MCP 集成

Dify 的定位是”AI 应用开发平台”——通过拖拽构建 Agent 工作流，配置模型、工具、知识库，然后部署为 API 或 Web 应用。这个模式对非技术人员友好，入门门槛极低。

根本性的范式冲突：Dify 的 Agent 是工作流中的一个节点；对于真正的 Agent 系统，Agent 应该是整个系统的入口，而不是流程图里的一格。

这个冲突在简单场景下不明显，但当你试图用 Dify 构建”意图驱动、开放推理”的 Agent 时，就会感受到这个范式的天花板。你会发现自己在用一个”确定性工作流工具”试图表达”不确定性推理过程”，这是两个本质上不同的计算模型。

Dify 的最大价值在于它覆盖了一个真实的市场需求：大量业务场景并不需要”真正的 Agent”，只需要一个可以调用 LLM 的自动化工作流。对这些场景，Dify 的低门槛和快速上线优势是实质性的。

适合：有固定流程的 AI 应用，RAG 增强问答，非技术团队快速上线

不适合：开放式推理 Agent，需要深度定制的领域系统

Coze / 扣子（字节跳动）

核心模式：全生命周期 Skill 管理 + 全链路 Tracing

Coze 最值得借鉴的不是它的 Agent 能力，而是它的全链路 Tracing 体系。

在 Coze 中，每次推理、每次工具调用、每个子任务都有完整的可观测链路。当 Agent 行为出现异常时，你可以精确定位是哪一步出了问题、当时的输入是什么、输出是什么。这在 Agent 系统调试和问题定位上是刚需，但很多框架把它当可选项而不是必选项。

全链路可观测性的价值，在 Agent 系统上比在传统 API 系统上更大。因为 Agent 的每一步都有不确定性，当行为出错时，你需要知道”是模型判断错了还是工具调用错了还是上下文理解错了”。没有全链路 Tracing，这个问题几乎无法系统性地回答。

其次是 Skill 生态：Coze 通过 Skill 市场形成了规模化的能力共享生态。这个理念证明了”过程性知识的共享”这个方向的市场可行性，对 Hermes 的 agentskills.io 和 AEGIS 等系统的设计有直接参考价值。

不适合：需要私有化部署的场景（闭源商业平台）

Hermes Agent（Nous Research）

Hermes 是目前开源 Agent 框架中产品形态最完整的，也是架构设计上最有原创思路的。关于 Hermes 的深度分析，请参考《Hermes Agent 架构分析与思考》一文，此处仅做对比维度的概要。

Hermes 的架构特色：从用户交互层（16 个平台）→ Agent 核心层（AIAgent + Tool Registry + Prompt Builder）→ 持久化与学习层（Session DB + Memory + Skills System）的三层架构，以封闭学习闭环（技能自创建 + 记忆固化 + 用户建模 + 跨会话搜索）为核心差异化。

Hermes 与其他框架的本质差异：其他框架的设计目标是”让 Agent 更容易创建”，Hermes 的设计目标是”让 Agent 随时间变得更好”。这是两个本质上不同的目标，决定了 Hermes 在架构选择上的很多取舍。

适合：个人 AI 助手，跨平台 Agent，学习闭环场景，RL 数据收集

不适合：多用户企业服务端（同步架构限制），需要强安全审计的场景

六、从 9 个框架提炼的 12 条设计理念

这是本文最有价值的部分——不是选哪个框架，而是从所有框架中学到什么。

编号设计理念来源核心价值
1Agent 循环引擎：LLM → 权限 → 执行 → 循环Claude Code同步推理骨架，每步内建检查点
2事件钩子体系（生命周期 15+ 事件）Claude CodeAOP 模式，不动内核做外围拦截
3上下文分层管理：Compaction + ToolSearchClaude Code断路器 + 工具三档加载
4五层纵深权限模型（多层递进评估链）Claude Code企业策略 → 信任 → Hook → 审计 → 确认
5MCP 工具协议（标准化工具接口）Claude Code + Dify工具标准化接口，跨系统互操作
6状态图 + 检查点恢复（任务暂停/恢复语义）LangGraph任务状态机管理，会话中断恢复
7Human-in-the-loop 一等公民LangGraph审批是”正常流程”不是”异常路径”
8角色化 Agent + Manager 专业化分工CrewAI上下文聚焦，工具权限分离
9事件驱动消息传递（结构化 task → conclusion）AutoGenAgent 间解耦，异步拓扑支持
10递归任务分解（深度自主规划）Hermes / 自研多步任务自主规划与执行
11Skill 市场架构（程序记忆可分发）Hermes / Coze过程记忆可复用、可分发、可共享
12全链路 Tracing（完整可观测链路）Coze每步推理和工具调用可追溯

来源分布的含义：Claude Code 贡献 5 条（最多），因为它是唯一经过日活百万级生产验证的 Agent 系统，其 Agent 循环 + Hook 体系 + 上下文管理的组合是目前工程成熟度最高的。LangGraph 贡献 2 条，CrewAI、AutoGen、Hermes、Coze 各贡献 1~2 条。每个框架都有值得学习的地方，只是学的是思想，不一定是代码。

七、一个被普遍忽视的维度：可观测性

在所有框架对比文章中，可观测性（Observability）几乎从未作为独立维度出现。但它恰恰是 Agent 系统从”可以运行”到”可以信任”的关键差距。

Agent 系统的可观测性挑战比传统分布式系统更复杂，原因在于不确定性的叠加：模型本身的不确定性（同样的输入可能有不同的输出）+ 工具调用的副作用（工具执行的结果可能影响后续推理）+ 多轮对话的上下文依赖（当前行为依赖于对历史上下文的理解）。

在这个背景下，一个无法被观测和追溯的 Agent 系统，在生产环境中是一个黑盒。当它出错时，你无法系统性地判断是模型问题、工具问题还是上下文问题，也无法通过重放来复现问题。

目前在这个维度上做得最好的是 Coze（全链路 Tracing 是核心产品能力之一）和 Claude Code（Hook 体系天然支持审计注入）。大多数编排库把 Tracing 当作可选配置，而不是架构约束。

如果你在设计生产 Agent 系统，建议把”完整的 Trace 链路”作为非功能需求写入设计文档，而不是在系统上线后再补救。

八、选型方法论：先问场景，再看框架

没有最好的框架，只有最合适的框架。在评估任何框架之前，先把这 4 个问题的答案确定清楚：

Q1：部署形态是什么？

如果是本地单用户工具，Claude Code / Codex CLI / Cursor 值得参考；如果是服务端多用户，LangGraph / CrewAI / AutoGen 或自研；如果是完整平台产品，Hermes / Dify / Coze。

Q2：工作流的确定性程度如何？

如果是固定流程图，LangGraph 的状态图 DAG 正好；如果是半结构化任务，CrewAI 的角色分工加有限编排能应对；如果是开放推理型 Agent，这些框架都会过度约束，需要自研循环引擎。

Q3：是否需要多 Agent 协作？

如果不需要，单 Agent 循环足够，不要引入不必要的复杂度；如果需要角色化协作，CrewAI 的理念正确但需要自研实现；如果需要异步分布式协作，AutoGen 的理念正确但注意 AG2 迁移风险。

Q4：安全与合规要求是什么级别？

如果无特殊要求，任意框架都可以；如果需要审计日志，要先确认框架是否原生支持而非可选插件；如果需要信任层级、多租户、合规审计，通用框架无法满足，需要自研引擎，设计模式从框架精华中提炼。

九、自研引擎的决策临界点

自研不是银弹，但在以下条件同时成立时，自研优于定制开源框架：

条件一：领域需求与框架设计目标根本冲突。不是”框架缺某个功能”，而是”框架的基本假设与你的场景不符”。比如框架假设单用户，而你的场景是多租户；框架假设确定性工作流，而你的场景是开放推理。

条件二：定制深度已经侵入框架内核。如果你发现自己在修改框架源代码而不是扩展它，这是一个强烈信号——你在和框架的设计决策对抗，而不是在使用它。

条件三：跟踪上游版本的维护成本不可忽视。尤其是当框架处于重构期（如 AutoGen 的 AG2 迁移），每次上游更新都需要你评估和适配，这个成本在长期维护中会累积。

条件四：团队有足够的设计参考。自研的最大风险是”重新发明轮子”，但有了充分的框架学习和失败案例分析，你在设计时有参照系，不是从零发明。

如果决定自研，建议的资源策略是：复用工具协议（MCP）、LLM 调用 SDK、持久化基础设施、可观测系统（OpenTelemetry）；提炼本文 12 条设计理念中符合场景的，不要全盘照搬；自研 Agent 循环引擎内核和领域特有的权限/审计/隔离机制。

十、总结：框架选型的本质是架构决策

框架选型表面上是技术问题，本质上是架构决策——你在决定哪些约束可以接受，哪些必须自己控制。

主流框架的发展轨迹有一个共同规律：从”Demo 友好”到”生产可用”之间，有一个巨大的工程鸿沟。大多数框架还停留在跨越这个鸿沟的途中。

Hermes 是目前开源框架中离这个鸿沟彼岸最近的——它有真实用户、有生产规模的验证（v0.9.0 已有 24 位贡献者、487 commits、16 个平台），但它的设计重心是”个人智能体”而非”企业服务端”，这个定位决定了它在多用户并发和安全合规方向上的天然局限。

Claude Code 是架构思想最值得学习的，但它的设计是为了让”一个开发者变得更强大”，不是为了让”一百个用户并发调用”。

真正的生产级 Agent 系统，今天基本上都是自研的。 框架提供的不是”可以直接用的产品”，而是”有价值的设计参考”。

这不是一个令人沮丧的结论——恰恰相反，这说明 Agent 基础设施还处于早期，有大量值得构建的空间。当 LLM 基础能力在快速提升时，Agent 工程的最大瓶颈不在于模型，而在于围绕模型构建的可靠性、可观测性、可审计性基础设施。这正是下一代 Agent 框架需要解决的核心问题。
